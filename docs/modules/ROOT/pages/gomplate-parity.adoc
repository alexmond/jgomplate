= gomplate Parity
:navtitle: gomplate Parity

This page tracks how much of upstream https://gomplate.ca[gomplate] jgomplate
implements — the namespaced function set, the datasource layer, and the CLI — and
flags where behaviour is only *partially* faithful so template authors know where
output may diverge from the Go `gomplate` binary.

[IMPORTANT]
====
jgomplate is an early *seed*. A working subset of the `strings`, `conv`, and `coll`
namespaces is wired end-to-end (engine + CLI); the bulk of gomplate's ~200-function
surface, the remote datasources, and most of the CLI are tracked follow-up work.
This page is the honest map of that gap — not a claim of completeness.
====

Baseline: gomplate v4 function namespaces as documented at
https://docs.gomplate.ca/functions/[docs.gomplate.ca]. Two function layers come
*for free* from the underlying gotmpl4j engine and are not re-counted here as
gomplate-specific work:

* **Go `text/template` builtins** — provided by `gotmpl4j-core` (priority 0).
* **Sprig functions** — provided by `gotmpl4j-sprig` (priority 100). gomplate also
  embeds the full Sprig library, so these render unchanged.

The gomplate *namespaces* (priority 200) are the jgomplate-specific layer, and are
where the parity gap lives.

== Datasource parity

gomplate's datasource system (`-d alias=<url>`, the `datasource`/`ds` functions,
`defineDatasource`, `include`) supports a long list of URL schemes with
`Content-Type`-based parsing. jgomplate currently reads *local files only*, by
extension.

[cols="1,1,3"]
|===
| Capability | Status | Notes

| Local `file://` JSON      | ✅ | Parsed by Jackson (`.json`).
| Local `file://` YAML      | ✅ | Parsed by the Jackson YAML dataformat (`.yaml` / `.yml`).
| Extension-based dispatch  | 🟡 | Format chosen by file suffix, *not* by `Content-Type` as gomplate does.
| TOML / CSV / ENV / plain  | ⬜ | Not parsed yet.
| `http` / `https`          | ⬜ | Remote fetch not implemented.
| `vault` / `consul` / `boltdb` | ⬜ | Not implemented.
| `aws+smp` / `aws+sm` / `s3` / `gs` | ⬜ | Not implemented.
| `merge:` (layered sources) | ⬜ | Not implemented.
| `datasource` / `ds` / `include` template funcs | ⬜ | The functions that expose a datasource *inside* a template are not registered; a loaded datasource is passed as the render context only.
|===

== CLI parity

The `jgomplate` command is config-driven: it builds a config from the flags actually
passed and overlays it on a `.gomplate.yaml` file (CLI wins), mirroring gomplate's
`Config.MergeFrom` precedence. It covers the input/output core plus config-file loading;
datasource, context, and directory machinery are follow-up work.

[cols="1,1,3"]
|===
| gomplate flag | jgomplate | Notes

| `-f, --file`          | ✅ `-f/--file` | Template file(s), repeatable; `-` = stdin.
| `-i, --in`            | ✅ `-i/--in`   | Inline template string.
| `-o, --out`           | ✅ `-o/--out`  | Output file(s), repeatable, paired with `-f`; `-` = stdout.
| `--config`            | ✅ `--config`  | Config file; precedence `--config` > `$GOMPLATE_CONFIG` > `.gomplate.yaml` (default-missing skipped, explicit-missing errors).
| `--experimental`      | ✅ `--experimental` | Parsed into config (no gated features yet).
| `-V, --verbose`       | ✅ `-V/--verbose` | Logs the resolved config file.
| `-d, --datasource`    | ✅ `-d/--datasource` | `alias=URL` reachable via `(ds "alias")` / `(datasource "alias" "path")` / `include`. Local file schemes (json/yaml); also the `datasources:` config key.
| `-c, --context`       | ✅ `-c/--context` | `alias=URL` bound to `.`; alias `.` merges into the root. Local file schemes (json/yaml); also the `context:` config key.
| `--input-dir` / `--output-dir` | ✅ | Recursive directory render, mirroring structure. Also `--output-map` (`.in` = input path), `--chmod`, and `--include`/`--exclude`/`--exclude-processing` globs (platform `glob:` — a faithful subset of gomplate's doublestar).
| `--exec-pipe`         | ⬜ | Not implemented.
| `-t, --template`      | ✅ `-t/--template` | `name=URL` registered as a partial, invoked via `{{ template "name" . }}`. Local file schemes; also the `templates:` config key.
| `--left-delim` / `--right-delim` | ✅ | Custom action delimiters (via gotmpl4j 1.2.2 `delims`); also `$GOMPLATE_LEFT_DELIM`/`$GOMPLATE_RIGHT_DELIM` and the `leftDelim`/`rightDelim` config keys.
| `--plugin`            | ⬜ | Config-file `plugins:` key parses; external-command execution not implemented.
| `--missing-key`       | ✅ `--missing-key` | `error` (default, matching gomplate), `zero`, `default`, `invalid`; also the `missingKey:` config key.
|===

NOTE: The full gomplate config-file schema (all `Config` keys) is parsed into the shared
`GomplateConfig` model today; keys whose behaviour is not yet wired (plugins, post-exec)
are accepted and ignored until their follow-up issue lands, so a gomplate `.gomplate.yaml`
loads without error.

[#known-limitations]
== Known engine limitations

* **Variadic namespace methods now work (gotmpl4j 1.2.1+).** Earlier versions matched
  methods by exact parameter count and did not unpack Java varargs, so a variadic
  method such as `math.Add(Object...)` resolved to nothing. gotmpl4j 1.2.1 unpacks
  varargs, so `Object...` namespace methods are callable directly. The varargs-unblock
  sweep is complete: `math` (`Add Mul Max Min Seq`), `coll` (`Slice Keys Values Dict
  Merge Sort Index Pick Omit Flatten`), `conv` (the `…s` plural forms), `strings`
  (`Indent RuneCount`), and `data` (`CSV CSVByRow CSVByColumn ToCSV`) are all done —
  no variadic function is blocked by the engine any more. The only functions still absent
  (`coll.JSONPath`, `data.CUE`) are blocked on an engine that has no faithful Java option,
  not a varargs fix.
* **`coll.JQ` matches; `coll.JSONPath` is deferred — the difference is dialect risk.**
  jq is one standardized language: gomplate's gojq and jackson-jq (what `JQ` uses here)
  implement the same semantics, so ordinary expressions agree. JSONPath is not: gomplate
  uses the Kubernetes dialect (`k8s.io/client-go/util/jsonpath` — leading-dot, brace-
  wrapped, k8s filters), which no mature Java library implements; Jayway/Goessner is a
  different dialect (`$`-rooted), so a Jayway-backed `JSONPath` would not match gomplate's
  expressions and is not shipped under the parity banner.
* **Namespace args arrive loosely typed** (`Object`) and are coerced through
  `functions.Values`. Coercion edge cases (e.g. numeric strings, nil handling) may
  differ from gomplate's Go type handling until each method has a parity test.
* **Non-deterministic functions** (once `crypto`, `random`, `uuid`, `time.Now` land)
  will not reproduce Go's exact byte output — only format/length/charset will be
  guaranteed, the same caveat jhelm documents for Sprig's random/crypto helpers.

== How to extend

Adding a gomplate function is a four-step change (see `CLAUDE.md` →
_Adding a gomplate function_):

. Find or create the namespace class in `jgomplate-functions/.../functions/ns/`.
. Add a `public` method named exactly as gomplate names it (PascalCase), taking
  `Object` params coerced through `Values`.
. If it is a *new* namespace, register it in `GomplateFunctionProvider#getFunctions`.
. Add a rendering test in `GomplateNamespaceTest` asserting against real engine
  output.

Keep method names verbatim to gomplate's Go API so templates written for gomplate
render unchanged.
